Spotify Playlist Project

November 15th, 2025

This project is a Python app built with Flet that generates random Spotify playlists using the Spotify Web API. The original version of this project was created by another developer, who had implemented the core functionality, but left significant usability and trust gaps. The onboarding process required the user to directly edit the Python script, and their credentials were stored in plaintext. On top of this, the UI didn’t offer any feedback and the interaction patterns were outdated and not aligned with modern application workflows.

Here is what the original version looked like:

My role in this project was to redesign the end-to-end experience, including improving security, reducing friction, and modernizing the UI. My goal was to make this feel like a real, user-friendly application rather than a script for a developer.

The Challenge:

The original app worked but was difficult to use and not intuitive. Users had to edit the Python script to authenticate, and the interface didn’t provide any feedback. The workflow was only accessible to developers and not regular users. The goal of this project was to make the application accessible and trustworthy, in order to make playlist generation accessible to a broader audience.

Problems and User Needs:

Onboarding Friction: The onboarding process involved the user having to open up the Python script and manually type in their credentials into the code. This required technical knowledge and added unnecessary friction. This means that there would be a higher chance that the user would have to go through lots of trial and error in order to authenticate. Secondly, the script also wanted the user to add a custom redirect URI, however, Spotify had recently standardized this, making it an unnecessary extra and confusing step.

Trust & Security: Since the user is required to enter the credentials into the plaintext of the script, they are not secure. Secondly, since the UI looks very outdated and does not have a good color scheme, this also lowers the user’s trust in the application.

Usability & Feedback: There is no support for securely storing the login, so if someone doesn’t want their credentials permanently stored in plaintext, they will need to enter them again every time they reopen the app. The app also doesn’t handle errors gracefully, and there are no indications of loading states in the app.

Nielsen's Heuristics: There is some debate in UX on whether it is necessary to do an analysis using Nielsen’s 10 Heuristics, however, I felt that this was valuable for this project because it allowed me to identify many ways in which the UI of this app needed improvement. These heuristics specifically made it so that it was easier for me to be able to identify issues related to error prevention, visibility of system status and user control. The following principles were violated:

  • Error prevention: No validation for number input
  • Visibility of system status: No loading indicators or feedback
  • Aesthetic & minimalist design: Very outdated interface
  • User control & freedom: No logout functionality
  • Recognition over recall: Credentials had to be retyped every time

Research:

Building an Interactive Login: To reduce friction in onboarding, I decided to modernize the sign-in process to make it intuitive. Research shows that unnecessary friction can also lower conversion (FullStory, 2021). The original version of this app stored credentials in plaintext, which is insecure (MITRE, 2006), and should instead be encrypted and stored in the keyring of the device (Read the Docs, 2025). To solve this, I designed an interactive login form that also removed the redirect URI (Spotify for Developers, 2025).

Designing an Emotionally Impactive UI: Users have varying preferences on whether they prefer dark mode or light mode (NNG, 2023). Allowing both improves accessibility, since some users find that light mode is too harsh. On top of this, color choices influence emotional perception, such as teal/aqua being associated with clarity and freshness, and dark backgrounds being associated with sophistication (Smashing Magazine, 2010). This allowed me to design the app’s theme, color palette, and support for dark/light mode. The visual appeal in the new design I made also makes the software perceived as easier to use, and since it looks better, users will be more tolerant of minor usability issues (NNG, 2024).

Providing Real-Time Feedback: Loading indicators and other methods of real-time feedback reduce uncertainty, prevent repeated actions and increase trust (Material Design, 2025). Based off of this, I incorporated spinners, feedback messages, and smoother transitions in the login and playlist generation.

Design Solutions:

Improving Visual Clarity: In the original, the UI looked outdated and consistent. Therefore, I made the typography larger, centered the form layout, made clear labels and used a consistent color scheme. These changes improved clarity and readability, and laid the foundation for a more trustworthy experience.

Building a Trustworthy User Experience: Originally, the app required users to enter their credentials in plaintext and go through lots of trial and error. In order to deal with this, I realized I had to focus on not only improving the security, but also the sense of trust, predictability and transparency. These were some of the solutions that I created in order to fix this:

  • Rounded corners and aesthetic UI.
  • Stores credentials in keyring, which improves security.
  • Automatically detects system light/dark mode.
  • This app is modern, which makes the user view it as a real application rather than some random script.
  • Shows spinners when things are loading, and doesn't freeze, so users know something is happening.
  • The flow is similar to real SaaS apps, which creates predictability.
  • The logout button allows users to delete all credentials and tokens, which makes the app feel more honest and gives the user more control.

These changes make it so that the app feels more professional, predictable and safe to use.

Making Playlist Generation Intuitive: In the original, the fields on the Playlist Generation page had pointed corners and provided no feedback to the user, so if there was an error, it would be very challenging for the user to debug. Secondly, once a playlist was generated, in order for the user to find this playlist, they would have to manually open up Spotify and go to the Playlist section, in which they would have to then search for their playlist. Therefore, I implemented rounded, styled text fileds, success/error messages, better labeling, spacing, and a clickable link to the playlist. Also, the app does not crash with invalid input. As a result, the playlist generation flow became more intuitive, errors are easier to fix, and users can immediately access their playlists.

Designing a Cohesive Color Scheme: Since the interface didn’t feel modern or expressive, I designed a color scheme that worked consistently across the light and dark theme.

Reflection:

I think that this application was significantly improved for users after I made the changes that I made, since the user experience is much more simplified, and the UI is polished.

Considering Other Edge Cases: If I were to revisit this project in the future, I think that it would be important to work on some other UX aspects, specifically in relation to potential edge cases. For example, the application assumes that the user already knows how to get their client ID, client secret and username from Spotify, however, some users could be confused at this part, which means that the application could benefit with an info toggle on the input fields.

Potential Accessibility Edge Cases: On top of this, I could have also added other settings that would have increased accessibility, such as the ability to toggle languages, manually switch between dark and light mode, or have touch ID authentication for Mac users. However, the constraint that stopped me from considering this at the time I made it was that I wanted to first make the application be very simple, in order to not overwhelm the user, and also provide visually aesthetic minimalism.

Constraints on Implementing These Solutions: In order to add all of these features, I would have had to consider how I would go about adding these features in the first place, and it would have either came from adding a settings button or just having the feature as a drop-down menu on the page itself. If there was a settings button, it would have required the user to click on it and go to another page, edit stuff in that page, and then go back, which could create mild friction. However, if settings such as manual dark/light mode toggle were on the page itself, this could also create clutter. My choice to keep the interface minimal did allow for a polished product with a strong UI, but it did also limit accessibility.

Building the Final Application: In order to create the designed experience, I built a fully functioning desktop app using Python and Flet. Even though this app is not production ready, I was able to test the authentication flows, loading feedback, error handling and the logic for creating playlists. Since I implemented the prototype myself, I intentionally focused on creating the redesigned experience. If I were to work in a cross-functional environment with an engineer, I would implement a wider range of features, such as more accessibility features and language toggles. For this project, my main focus was on clarity and UX validation.

Constraints of Flet: In the end of the experience, when a playlist is generated, there is a clickable link to that playlist that appears in the UI that says ‘Open Playlist’, which allows the user to view the playlist on Spotify. It would have been better if this app allowed the user to preview the playlist in the app itself, so the user wouldn’t have to open any link, which could break their flow. However, I found out when I got to this part that Flet doesn’t allow this to happen, meaning that there was no way to actually program this into the application. Because of this, the only way to add this feature would be to rewrite the entire application using a different module than Flet, or even switching to a different language besides Python.
